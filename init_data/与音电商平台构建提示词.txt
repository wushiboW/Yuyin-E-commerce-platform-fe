### 0.4 并发开发物理隔离方案：Git Worktree

#### 背景与目标
在同时操控 20+ 个子服务进行并行开发时，多线程或多会话如果直接操作同一目录，极易引发文件并发写入冲突、状态不一致等问题。为此，我们采用 **Git Worktree** 方案，为每个子服务创建**独立的工作树**，实现文件系统级别的物理隔离，同时仍共享同一个 Git 仓库的历史和对象存储。

#### 方案细节
1. **主仓库初始化**  
   项目根路径即为 Git 仓库根目录。若尚未初始化，你需首先执行：
   ```bash
   cd /Users/wushibo/some_code/BE/company_projects/Yuyin/Yuyin-E-commerce-platform
   git init
   git add -A
   git commit -m "Initial project skeleton"

2.为每个子服务创建独立 Worktree
每个子服务（按《微服务子应用定义与设计表》中的清单）都拥有一个独立的工作目录，基于一个独立的开发分支（命名建议：dev/<service-name>）。示例如下：
# 假设子服务名为 user-service
git worktree add ../yuyin-user-service -b dev/user-service main
实际执行时，请根据子服务定义表，在父目录 /Users/wushibo/some_code/BE/company_projects/Yuyin/ 下为每一个子服务创建对应的 worktree，目录名统一采用 yuyin-<service-slug> 格式。每个 worktree 都会自动检出基于 main 的新分支，且物理上完全独立。

3.开发时的目录切换
当你需要开发某个子服务时，必须切换到该子服务的 worktree 目录进行操作。例如开发用户服务时，当前工作目录应为：
/Users/wushibo/some_code/BE/company_projects/Yuyin/yuyin-user-service
在你的主显示器 Ghostty 分屏中，每个子屏幕的会话也应将其工作目录（cwd）定位到对应的 worktree。

4.代码提交流程

在每个 worktree 中独立进行 git add、git commit，改动仅记录在当前分支。

禁止在一个 worktree 中直接编辑另一个 worktree 的目录，跨 worktree 的文件拷贝也必须通过 Git 分支合并完成。

定期将各开发分支合并回 main（由架构师或项目管理 Agent 在合适时机协调）。合并前，请确保当前迭代批次所有服务均已提交。

5.共享模块的处理
所有 worktree 中都包含完整的项目目录结构（包括 packages/shared 等公共模块），但它们的文件内容初始一致。如果你需要修改共享模块（如 Protobuf 定义、通用类型），应选择一个专门的 worktree（或主仓库）进行修改，然后通过 git merge main 将变更同步到其他 worktree。严禁在多个 worktree 中同时修改同一共享文件。

6.管理命令备忘
查看所有 worktree：git worktree list
删除某个 worktree（开发完成后）：git worktree remove ../yuyin-user-service
清理失效的 worktree 记录：git worktree prune


7.重要约定

任何时候，一个 worktree 只能检出一个分支。因此每个子服务必须使用独立的 dev/<service-slug> 分支，且严禁在两个 worktree 中检出相同分支。
在 Ghostty 的每个 Tab 或子屏幕中，请通过环境变量或提示符清晰显示当前 worktree 的服务名称，避免误操作。

## 四、后端架构设计

### 4.1 多语言混合架构原则
依据不同业务场景的技术特性，选择最合适的编程语言与框架，实现优势互补：
	•	Python：业务逻辑复杂、需求多变、AI/ML 生态成熟。负责核心交易链路与全部 AI 中台。
	•	Rust：极致性能、内存安全、无 GC 抖动。用于库存、风控、API 网关等对延迟和吞吐要求苛刻的模块。
	•	Go：高并发、轻量协程、部署简单。承接搜索、推荐、消息推送等 IO 密集与长连接场景。
	•	Java：生态丰富、稳定性强、企业级集成成熟。用于物流、供应链等需对接大量第三方且业务持久的模块。
	•	C++：计算密集型、硬件级图像/视频处理性能无可替代。负责图像处理与视频转码。
	•	C#：企业级数据处理与 Windows 生态集成。负责财务对账与报表。
	•	C：最底层安全与硬件交互。用于加解密服务，直接调用硬件加密机 (HSM)。

### 4.2 Python 开发环境
- 所有 Python 服务在对应模块路径下使用 **uv** 创建虚拟环境进行开发
- 示例：`uv venv && uv pip install -r requirements.txt`

### 4.3 微服务拆分原则
- **领域驱动设计（DDD）**：按核心域、支撑域、通用域划分，避免数据耦合
- **单一职责**：每个服务只负责一个业务聚合根，独立可部署
- **去状态化**：服务本身无状态，状态外移至 Redis 或 DB，方便水平伸缩

### 4.4 数据一致性方案
核心交易链路（下单→锁库存→支付）采用 **TCC/Saga 混合分布式事务**：
- 订单服务作为 Saga 协调器
- 库存服务提供 Try-Confirm-Cancel 接口
- 支付服务提供正逆向接口
- 全部失败则触发补偿（取消订单、释放库存、退款）
- **最终一致性**：订单支付成功后的事件驱动下游（物流、积分、推荐），Kafka 保证异步可靠投递

### 4.5 缓存策略
- **Cache-Aside 模式**：读取先查缓存（Redis），未命中查 DB 并回填；更新先写 DB，再删缓存
- **库存热点**：Redis Lua 原子操作，配合 Rust Actor 内存缓存

### 4.6 高可用与性能
- **HPA 自动扩缩**：重点服务（库存Rust、订单FastAPI、搜索Go、购物车FastAPI）基于 CPU/内存/自定义指标弹性伸缩
- **限流熔断**：网关层 Pingora 令牌桶；gRPC 超时 2s，失败重试 2 次
- **降级**：营销/推荐故障时返回静态默认数据；风控失败时放行（需人工监控）
- **容灾**：跨可用区部署，Patroni+etcd 实现 PostgreSQL 自动故障转移

### 4.7 安全设计
- **API 网关**：统一 SSL/TLS 卸载、JWT 验签、IP 黑名单、防重放攻击
- **服务间认证**：gRPC 使用 mTLS 双向认证（Istio 或自签证书）
- **敏感数据**：用户密码 BCrypt 加盐哈希；支付信息、手机号脱敏存储
- **密钥管理**：C 服务操控硬件加密机，其他服务不可直接接触密钥
- **风控**：登录频次限制、异地登录检测、下单行为分析

### 4.8 可观测性
- **监控**：Prometheus 采集全服务指标 → Grafana 统一仪表盘
- **日志**：结构化 JSON 日志 → Fluentd 收集 → ELK，TraceId 关联同次请求
- **链路追踪**：Jaeger + OpenTelemetry，gRPC 拦截器自动传播 Trace Context，全语言 SDK 集成

### 4.9 AI 中台集成
- AI 推理中台作为**独立服务**，避免模型加载/推理阻塞业务
- **模型管理**：MLflow 训练产出 → 推送至 MinIO → 中台动态加载
- **特征平台**：Feast 支撑在线推理实时特征，避免特征-模型不一致
- **AIGC 异步**：Celery 任务队列，避免阻塞 API，生成结果回调通知
